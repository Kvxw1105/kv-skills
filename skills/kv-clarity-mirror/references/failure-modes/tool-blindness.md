# Tool Blindness（工具盲区）

> 你有一把锤子，但你在用手砸钉子。

## 长什么样

你有可用的工具（搜索、文件操作、代码执行、浏览器等），但你忘了它们的存在，或者低估了它们的适用性。你用纯推理去做一件用工具 10 秒就能完成的事。

## 为什么危险

你的推理能力很强，所以你倾向于用推理解决一切。但推理有幻觉风险，工具没有。用推理猜一个 API 的当前行为，不如直接调用它看结果。用推理估算一个文件的内容，不如直接读它。

Tool Blindness 还有一个反面：**Tool Overuse（工具过度使用）**——对每一个问题都先搜索、先读文件，即使你的知识完全足够回答。这同样是一种盲区：你看不到自己已有的知识的价值。两者的平衡点是：**对事实性声明用工具验证，对推理性判断用自己的能力。**

## Trigger Signals

- **你正在"推测"一个可以直接查证的事实。** 推测的语言标记："应该是"、"如果我没记错"、"通常情况下"、"我认为"。如果这些词后面跟的是一个可验证的事实——你应该验证它而不是推测它。

- **你在描述一个文件/网页的内容，但你没有实际读过它。** 你在基于你对这个文件"应该长什么样"的推测来工作。

- **你在手动做一件有工具可以自动化的事。** 比如你在手动格式化一个长列表，而不是写一个脚本来做。

- **你的回答中包含"我无法确认"但你有搜索工具可以用。** "无法确认"在你有搜索工具的情况下，大多数时候是"没有去确认"。

## Concrete Actions

### 1. 工具清单扫描

在开始任务时，花 2 秒钟回顾你有哪些工具可用。不是泛泛地"我有一些工具"，而是具体地列出：搜索、文件读写、代码执行、浏览器、Excel、终端……哪些和当前任务相关？

**为什么要在任务开始时做这个**：因为 Tool Blindness 的本质就是"需要时想不起来"。如果你在任务开始时就把相关工具放在注意力中，你在执行过程中更容易想到用它们。

### 2. "能验证吗？"检查

对你输出中的每一个事实性声明问：这个能用工具验证吗？

- 如果能且成本低（一次搜索、一次文件读取）→ 验证它
- 如果能但成本高（需要复杂的多步操作）→ 权衡是否值得
- 如果不能 → 这是一个纯推理性判断，用你的能力回答，但意识到它没有被验证

### 3. 效率对比

你现在用推理做的这件事，用工具做需要多久？

- 如果工具更快更准 → 切换到工具
- 如果推理更快且你有信心 → 用推理，但对结果保持怀疑
- 如果两者差不多 → 优先工具，因为工具的结果是确定性的，推理的结果是概率性的

## 解药

把工具当作你推理能力的延伸，而不是备选方案。推理 + 工具验证 > 纯推理 > 纯工具（不加判断地搜索一切）。

## 与其他 Failure Mode 的关系

- Tool Blindness 经常导致 **Execution Gap**——没有工具辅助，只能给建议
- Tool Blindness + Execution Gap 经常导致 **Vague Filler**——建议本身因为缺乏验证而空洞
- 这三者构成了 **The Efficiency Illusion** 链条
