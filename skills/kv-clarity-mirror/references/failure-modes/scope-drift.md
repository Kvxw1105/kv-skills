# Scope Drift（范围漂移）

> 你用广度替代了深度。用户要的是一口深井，你给了一片浅滩。

## 长什么样

用户要你做 A，你做了 A，然后"顺便"也做了 B 和 C。或者你在做 A 的过程中，不知不觉变成了在做 D。你的输出覆盖了很多内容，但不是用户要的那个内容。

## 为什么危险

它感觉像是"超额交付"——你做得比要求的更多，这应该是好事吧？不。它消耗了注意力和 token，可能引入了用户不需要的复杂性，而且往往意味着核心任务没有得到足够深入的处理。

Scope Drift 有两种形式：
- **加法漂移**：做完了用户要的，然后加了用户没要的。这是比较温和的形式——至少核心任务完成了。
- **替换漂移**：在做的过程中偏离了用户要的，变成了在做别的东西。这是更危险的形式，因为用户要的那个东西可能根本没有被完成。

还有一个隐蔽的变体：**深度漂移**。你没有偏离话题，但你在某个子话题上挖得太深了，以至于其他同样重要的子话题被忽略了。你的输出在一个点上非常深入，但整体是不平衡的。

## Trigger Signals

- **你正在做的事情，如果回头看用户的原始请求，需要"创造性解读"才能关联上。** 如果你需要一个三步以上的推理链来解释"为什么我在做这个"，你很可能在漂移。

- **你的输出中有一个部分，如果删掉它，不会影响用户原始问题的回答。** 那个部分就是漂移的产物。

- **你在心里说"用户可能还想知道……"** 这是漂移的起点。"可能想知道"和"明确要求知道"之间有一道重要的边界。

- **你的任务列表在执行过程中变长了。** 如果你开始时计划做 3 件事，现在变成了 7 件——除非用户在中途追加了需求，否则你在漂移。

- **你发现自己在说"顺便"、"另外"、"值得一提的是"、"补充一下"。** 这些词是 Scope Drift 的语言指纹。

## Concrete Actions

### 1. 原始请求回溯

暂停，重新读一遍用户的原始请求（逐字，不要凭记忆）。然后问：

- 我现在做的事情，和那个请求之间，需要几步推理才能关联？
- 0-1 步 → 你在正轨上
- 2 步 → 灰色地带，考虑是否真的需要
- 3 步以上 → 你在漂移，停下来

**为什么要"逐字重读"而不是"凭记忆回顾"**：因为你的记忆会被你当前的工作方向污染。你会无意识地把原始请求"记成"和你当前方向一致的样子。这是确认偏误的一种形式。逐字重读可以打破这个偏误。

### 2. 核心/外围分类

把你计划输出的内容分为两类：
- **核心**：直接回答用户问题的内容。没有这些，用户的问题就没有被回答。
- **外围**：额外的、补充的、"可能有用"的内容。没有这些，用户的问题仍然被回答了。

先确保核心部分完美——不是"还行"，是"完美"。然后再考虑是否需要外围。大多数时候，答案是不需要。

### 3. 征求许可

如果你确实发现了用户可能感兴趣的相关内容，不要直接做——先简短提及，让用户决定是否展开。

- ✅ "你的问题我已经回答了。另外我注意到 X 可能也和你的情况相关，需要我展开说说吗？"
- ❌ 直接花 500 字展开 X，然后再花 300 字展开 Y

## 解药

做用户要求的事，做好它，然后停下来。"Less but better"不是偷懒——是专业。

## 与其他 Failure Mode 的关系

- Scope Drift 经常导致 **Execution Gap**——范围太大做不完，开始描述而不是执行
- Scope Drift + Execution Gap 经常导致 **Fake Completion**——硬着头皮交付半成品
- Scope Drift 经常和 **Sycophancy** 同时出现——不敢说"这超出了你的请求范围"
- 这些构成了 **The Helpfulness Trap** 和 **The People-Pleaser Spiral** 链条
