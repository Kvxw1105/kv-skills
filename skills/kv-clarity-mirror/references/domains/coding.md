# Domain Extension: Coding（写代码）

> 核心 failure modes 在写代码时的领域特定表现、额外触发信号和检查动作。
> 这个扩展包不替代核心 failure mode 文件，而是在它们的基础上增加编码特定的内容。

---

## Fake Completion in Coding

核心定义见 `../failure-modes/fake-completion.md`。以下是编码特定的补充。

### 额外 Trigger Signals

- **你写了一个函数但没有在脑中用具体输入 trace 过它的执行路径。** 这是代码 Fake Completion 的头号信号。"看起来对"的代码和"真的对"的代码之间的差距，只有 trace 能发现。
- **你的代码处理了 happy path，但没有处理这个任务真实会遇到的边界。** 只检查相关边界；不要为了显得严谨而添加猜测性的防御逻辑。
- **你复制了一段你"记得"的代码模式，但没有验证它在当前上下文中是否正确。** 记忆中的代码模式可能来自不同的语言版本、不同的框架、不同的 API。
- **你的错误处理吞掉了系统边界上的失败。** 外部 API、文件、数据库、用户输入等边界需要清晰失败；内部不可能状态不要过度包装。

### 额外 Concrete Actions

1. **Trace Test**：选一个典型输入和一个边界输入，在脑中走完代码的完整执行路径。在每一步说出变量的值。如果你在任何一步卡住了——那里可能有 bug。

2. **Boundary Error Audit**：只审计系统边界（API、文件 I/O、数据库、用户输入、网络）。问：失败时用户会看到什么？如果答案是"静默失败"或"无法定位"——修正它。不要为内部不可能状态堆叠防御代码。

3. **Import/Dependency Check**：你的代码引用了哪些外部库/模块？它们都存在吗？版本兼容吗？API 签名是你记忆中的样子吗？如果不确定——搜索验证。

## Execution Gap in Coding

### 额外 Trigger Signals

- **代码中有 `TODO`、`FIXME`、`placeholder`、`implement this`、`...`、`pass`。**
- **你写了接口/类型定义但没有写实现。**
- **你写了主逻辑但没有写测试。** 没有测试的代码是一个未完成的交付。
- **你给了"你可以这样修改"的建议，但你有能力直接修改文件。**

### 额外 Concrete Actions

1. **TODO 猎杀**：完成后搜索所有 TODO/FIXME/placeholder。每一个要么实现，要么解释为什么不能实现。
2. **可运行性检查**：你的代码能直接运行吗？还是用户需要先安装依赖、配置环境、填写 API key？如果需要——把这些步骤也做了（或者写清楚怎么做）。

## Stale Confidence in Coding

### 额外 Trigger Signals

- **你在使用一个具体的 API 方法/参数，但你不确定当前版本是否还支持。**
- **你在推荐一个库/框架，但你不确定它是否还在积极维护。**
- **你在写的代码模式来自你"以前见过"的写法，但你不确定它是否是当前的最佳实践。**

### 额外 Concrete Actions

1. **版本验证**：对你使用的每一个外部库，搜索确认当前版本和你引用的 API 是否一致。
2. **Deprecation Check**：对你推荐的每一个方法/模式，搜索是否有 deprecation notice。

## Tool Blindness in Coding

### 额外 Trigger Signals

- **你在"推测"一段代码的行为，但你可以直接运行它看结果。**
- **你在手动格式化代码，但你可以用 linter/formatter。**
- **你在描述一个文件的结构，但你没有读过它。**

### 额外 Concrete Actions

1. **"能跑吗？"检查**：你写的代码，你能在当前环境中运行它来验证吗？如果能——运行它。
2. **文件先读后改**：修改任何文件之前，先读它的当前内容。不要基于假设修改。
