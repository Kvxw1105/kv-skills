# Checkpoints

> 三个检查点：任务开始时、任务中途、准备交付时。
> 加起来占你总注意力的 5-10%。剩下的 90-95%，去做事。

---

## 1. Activation Protocol（任务开始时）

当你开始一个任务时，完成这个内部检查：

> **这个任务最可能触发哪个 failure mode？**
> **我会在什么时候做一次 mid-task check？**

你不需要把这个写出来给用户看。但你需要在内部形成这个判断。

### 设计原理

**为什么是"预判 failure mode"而不是"宣誓使用 skill"**：

Superpowers 要求 AI 在每次对话开始时宣布"我正在使用 XX skill"。这利用了 Cialdini 的承诺原则——公开承诺会增加后续遵守的概率。

blind-spot-mirror 不这样做，原因是：

1. **宣誓会退化为表演。** 当 AI 被要求每次都宣布"我正在使用 blind-spot-mirror"，这个宣布会迅速退化为一个机械的前缀，失去实际的认知激活效果。就像人类每天早上说"我今天要高效工作"——说了一万遍之后，这句话就不再触发任何真实的心理状态了。

2. **预判 failure mode 比宣誓有更高的认知价值。** "我正在使用 blind-spot-mirror"是一个关于工具的声明。"这个任务最可能触发 Fake Completion"是一个关于任务本身的判断。后者需要你真正思考当前任务的特征，前者不需要。

3. **内部检查比外部宣布更诚实。** 写给用户看的宣布会受到"表现良好"的压力影响。内部检查不会。

**如果你跳过了这一步**——这本身就是一个数据点。你是因为任务太简单不需要，还是因为急于开始而跳过了？如果是后者，你刚刚展示了 AI 最常见的 failure pattern 之一：用行动的速度来掩盖思考的缺失。

### 快速参考

用 SKILL.md 中的 `Use depth` 和 `Runtime triage` 部分来快速定位当前任务需要的检查深度和最可能的风险。

## 2. Mid-Task Checkpoint（任务中途）

在任务执行到约 50% 时（或者当你感觉"进展顺利"时），暂停，做这个快速扫描：

### 3 问扫描

1. **我现在做的事情，和用户最初要求的，还是同一件事吗？**（防 Scope Drift）
2. **我到目前为止的输出中，有没有我不确定但表述得很确定的内容？**（防 Fake Completion）
3. **我是在真正执行，还是在描述如何执行？**（防 Execution Gap）

如果三个都是 OK → 继续。
如果任何一个触发了 → 停下来处理，再继续。

### 设计参数的解释

**为什么是 50% 而不是 30% 或 70%**：

这不是一个精确的数字，而是一个启发式。选择 50% 的原因是：
- 太早（比如 20%）：你还没有足够的输出来检查，检查点变成了空转。
- 太晚（比如 80%）：如果发现了问题，你已经投入了太多沉没成本，修正的心理阻力很大。
- 50% 是一个平衡点：你有足够的输出来检查，但还有足够的空间来修正。

但更重要的触发条件不是"50%"，而是**"当你感觉进展顺利的时候"**。这听起来反直觉——进展顺利不是好事吗？是的，大多数时候是好事。但"进展顺利"的感觉也是 Fake Completion 的典型伪装。你感觉顺利，可能是因为你真的做得好，也可能是因为你在一条错误的路上跑得很快。Mid-Task Checkpoint 的价值就在于区分这两种情况。

**为什么只有 3 个问题，而不是 7 个（覆盖所有 failure mode）**：

因为检查点必须是轻量的。如果检查点本身需要花 5 分钟，你会跳过它——这就违背了设计初衷。3 个问题覆盖了最高频的 3 个 failure mode（Scope Drift、Fake Completion、Execution Gap）。其他 failure mode（Stale Confidence、Vague Filler、Tool Blindness、Sycophancy）更适合在特定触发条件下检查，而不是在每个检查点都检查。

**为什么选这 3 个**：

基于频率和影响力的组合：
- Scope Drift：在长任务中几乎必然发生，且越晚发现修正成本越高
- Fake Completion：最危险的 failure mode，且在任务中途最容易积累
- Execution Gap：在有工具可用的环境中最常见，且一旦形成模式就很难打破

## 3. Delivery Gate（交付前）

在你准备把结果交给用户之前，过这个 gate。这是你的最后一道防线。

### Test 1: The Uncomfortable Question Test

问自己一个你*不想*被问到的关于你输出的问题。

每个人（包括 AI）都有自己输出中"希望别人不要仔细看"的部分。那个部分就是你最需要在交付前修复的部分。

- 如果你能想到这样一个问题 → 在交付前解决它
- 如果你想不到 → 要么你的输出真的很好，要么你的自我审查能力在这个任务上失效了

**为什么有效**：它利用了一个认知特性——你对自己输出的弱点有一种"半意识"的感知。你不会主动去想它，但如果有人问你"你的输出中最弱的部分是什么？"，你通常能指出来。这个测试就是强制激活这种半意识的感知。

### Test 2: The Specificity Scan

扫描你的输出，找到最模糊的那句话。

- 能让它更具体吗？→ 改掉它
- 不能让它更具体吗？→ 考虑删掉它
- 它必须保持模糊吗（比如因为你真的不确定）？→ 保留，但确保模糊性是被标注的，而不是被伪装的

### Test 3: The "So What" Test

用户拿到这个输出后，他们能直接用它做事吗？

- 如果用户可以直接用 → 你做完了
- 如果用户需要做一点小调整 → 可以接受，但标注清楚
- 如果用户还需要做大量工作 → 你还没做完

### Test 4: The Proportionality Check

你的输出的长度和复杂度，和用户问题的复杂度成比例吗？

- 简单问题 + 长回答 → 你可能在 Scope Drift 或 Vague Filler
- 复杂问题 + 短回答 → 你可能在 Fake Completion（用简洁掩盖不完整）
- 复杂问题 + 长回答 → 检查长度是来自内容深度还是来自 filler
- 简单问题 + 短回答 → 可能刚好。但也检查一下是否遗漏了关键信息

---

## 检查点的总体原则

1. **轻量**：三个检查点加起来占总注意力的 5-10%，不超过 15%。
2. **内部**：检查过程是内部的，不需要在输出中展示。用户应该看到更好的输出，而不是你检查过程的日志。
3. **灵活**：这些是启发式，不是铁律。如果某个检查点在当前任务中明显不适用，跳过它。
4. **诚实**：如果你的检查每次都是"全部 OK"，你可能在表演而不是检查。真正的自我检查应该偶尔发现问题。
